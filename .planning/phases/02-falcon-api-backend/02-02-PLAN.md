# Plan 02-02: Add API smoke verification and error/empty-state handling

<objective>
Harden the Falcon API from plan 02-01 with explicit invalid-filter responses, stable empty-state payloads, skipped-file diagnostics, JSON-native response checks, import-boundary checks, and a reusable ad-hoc smoke verification script using Falcon testing utilities. This plan depends on the API package created by 02-01 and must verify without launching a long-running server. Schema push gate result: no schema/ORM files are relevant to this Python/Falcon/CSV phase, so do not add schema push or migration tasks.
</objective>

## Artifacts this phase produces

- `scripts/verify_falcon_api.py` — focused ad-hoc verification script using `falcon.testing.TestClient(create_app())`; includes endpoint smoke checks, invalid-filter checks, JSON-native checks, and import-boundary scans.
- Hardened `inflation_dashboard/api/filters.py` symbols: `ApiFilterError`, `ParsedFilters`, `parse_common_filters`, `parse_bool_param`, `load_filtered_history`, and validation branches for unknown retailers, bad dates, reversed dates, negative max files, explicit all-history, invalid limits, and selected-file metadata.
- Hardened `inflation_dashboard/api/resources.py` behavior for stable 400 error envelopes and stable 200 empty result shapes across `HealthResource`, `InventoryResource`, `HistoryResource`, `RetailerAveragesResource`, `MoversResource`, and `CoverageResource`.
- Endpoint smoke coverage for `/api/health`, `/api/inventory`, `/api/history`, `/api/retailer-averages`, `/api/movers`, and `/api/coverage`.
- Boundary verification rules: API modules may import `falcon` but not `streamlit`, `streamlit_app`, or `plotly`; `inflation_dashboard/domain/*`, `inflation_dashboard/application/*`, and `inflation_dashboard/adapters/csv_price_repository.py` must not import Falcon or Streamlit.

<task id="02-02-T1" type="hardening">
<name>Complete invalid-filter validation and HTTP 400 envelope handling</name>
<read_first>
- `inflation_dashboard/api/filters.py` from plan 02-01.
- `inflation_dashboard/api/resources.py` from plan 02-01.
- `inflation_dashboard/api/serialization.py` from plan 02-01.

- `.planning/phases/02-falcon-api-backend/02-PATTERNS.md` error and empty-state patterns.
</read_first>
<action>
Harden `filters.py` and `resources.py` so invalid query inputs consistently return HTTP 400 with `error_envelope("invalid_filter", <short English message>, meta=<raw or partial filters>)`. Cover unknown `retailer`, bad `start_date` or `end_date` format, `start_date > end_date`, negative `max_files`, invalid `all_history` boolean values, unsupported `aggregation` outside `Average`/`Median`, invalid `limit`/`mover_count` outside 5..30 for movers, and invalid `category_limit` less than 1 for coverage. Ensure every resource catches `ApiFilterError` or equivalent validation errors and sets `resp.status = falcon.HTTP_400` with the envelope. Keep messages concise and deterministic enough for frontend display later.
</action>
<acceptance_criteria>
- Unknown retailer requests return HTTP 400 with top-level keys `data`, `meta`, and `errors`, `data` set to null, and at least one error object with `code: "invalid_filter"`.
- Invalid dates and reversed date ranges return HTTP 400 envelopes rather than tracebacks.
- `max_files=-1` returns HTTP 400; `max_files=0` remains valid and means uncapped.
- Unsupported `aggregation=Mode` on `/api/retailer-averages` returns HTTP 400.
- `limit=4` or `limit=31` on `/api/movers` returns HTTP 400 if the chosen range is 5..30.
- Error message strings are short English messages and do not include Python tracebacks.
</acceptance_criteria>
<verify>
Run `python - <<'PY'` from the repo root with Falcon `TestClient(create_app())` if Falcon is installed, exercising invalid retailer, invalid aggregation, negative `max_files`, and reversed dates. Assert each response status is 400 and each JSON body has envelope keys plus non-empty `errors`. If Falcon is not installed, run source assertions for the validation branches and record the dependency blocker for the final smoke script.
</verify>
</task>
